Database Decision
Why MongoDB Only — No PostgreSQL

One database, one driver, one mental model. Here's why it's the right call for this system.

❌ Why PostgreSQL Was Overkill Here

PostgreSQL shines for complex relational joins, transactions, and strict schemas. But WeLe's data is naturally document-shaped — a room has its blocked topics embedded, a user carries their violation history, a boundary rule has nested keyword arrays. Mapping this to SQL tables creates unnecessary JOIN complexity with no real benefit for a chat platform.

✅ Why MongoDB Is Perfect Here

Every entity in WeLe maps cleanly to a document. Rooms embed their own blocked_topics array. Users embed violation history. Boundaries embed keyword arrays. You get a single Mongoose driver for everything, Atlas free tier covers all your storage needs, and queries are fast with proper indexes. For a hackathon, one DB = less config, less bugs, faster build.

✅ Mongoose Does Everything pg Did

All the auth logic, room management, boundary rules, room requests — everything that was in PostgreSQL tables is now a Mongoose model. You write schema validation in JavaScript instead of SQL DDL. Relationships are handled via ObjectId references or embedding. The Node.js code gets simpler, not more complex.

✅ MongoDB Atlas Is Perfect for Hosting

MongoDB Atlas free tier gives you a 512MB shared cluster with a connection string you paste straight into your .env file. Render + Atlas + Upstash (Redis) = your entire backend stack on free tier. No credit card required for any of them during a hackathon.

💡 Denormalization Note
In MongoDB it's normal and recommended to store username inside the Message document even though it also exists in the User document. This avoids a JOIN ($lookup) on every message fetch. Same for storing user_name in RoomRequest. This is a MongoDB best practice — not a design flaw.

PHASE 2
Python AI Moderation Engine — 5-Layer Pipeline
~90 mins · The most important piece. Get this right first.

Pipeline Architecture

POST /validate ← called by Node.js chat server (server-side, not client) body: { message: str, room_id: str, user_id: str } Layer 1 — PII + Regex (5ms) → Indian phone (10 digits), email, URLs, Aadhaar, PAN, passwords → If match: BLOCK immediately Layer 2 — Generic Keyword Match (10ms) → Load from Redis (refreshed from MongoDB every 5 min) → Check each enabled boundary category's keywords → If match: BLOCK with category feedback Layer 3 — Toxicity (60–100ms) → Detoxify model, block if score > 0.7 Layer 4 — Topic Similarity (100–200ms) → Fetch room's blocked_topics from MongoDB → SBERT embeddings → cosine similarity → Block if any blocked topic similarity > 0.65 Layer 5 — LLM Groq (200–400ms) [only if layers 1-4 pass] → Groq llama-3.1-8b-instant → Returns ALLOW/BLOCK + reason + improved suggestion → Skip if message clearly fine (confidence optimization) Response: { status, reason, feedback, suggestion, layer_triggered }

✅ MongoDB vs PostgreSQL — One Key Difference in Your Code
MongoDB uses _id (ObjectId), not id (integer). When passing room_id from Node.js to Python AI service, always do room_id.toString(). In Python, wrap it: from bson import ObjectId and use ObjectId(room_id_string) for MongoDB queries. This is the #1 source of bugs when mixing Node.js ↔ Python ↔ MongoDB.
💡 Groq Free Tier Limits — Plan Around This
Groq free tier: ~14,400 requests/day on llama-3.1-8b-instant. For a hackathon demo that's plenty. However Layer 5 (LLM) should only trigger for ambiguous messages — add a quick pre-check: if message length < 10 chars or passes all 4 previous layers confidently, skip L5. This saves API calls and reduces latency.
⚠️ ML Models RAM on Free Hosting
Detoxify multilingual (~300MB) + SentenceTransformer all-MiniLM-L6-v2 (~90MB) + spaCy (~50MB) = ~440MB. Render free tier has 512MB RAM. It's tight. If it crashes: drop Detoxify and use Groq (L5) for toxicity too — just ask Groq to also check toxicity in its system prompt. That reduces RAM to ~150MB.
